自动化雪碧图生成 & 现代方案对比

手工拼图不可持续，真实项目必须依赖构建工具自动生成雪碧图。 本节演示自动化工具链、四种图标方案的横向对比，以及 HTTP/2 时代的现代替代方案。

1 自动化雪碧图生成工具

手工维护雪碧图极易出错（坐标算错、漏图、命名冲突），生产环境必须用工具自动完成 "源图 -> 合并 -> 生成 CSS" 流程。以下是主流工具：

webpack-spritesmith
Webpack 插件，将指定目录下的图片自动合并为雪碧图并生成 SCSS/CSS/JSON。
Webpack 维护缓慢
sprity
Node 通用雪碧图生成器，支持 Gulp/Grunt，输出格式可定制。
Gulp 已停止维护
postcss-sprites
PostCSS 插件，扫描 CSS 中的 background-image 自动替换为雪碧图。
PostCSS 低频维护
Glue
Python 命令行工具，支持 SVG 输出、Retina、缓存等特性。
CLI 已停止维护
@vitejs/plugin-spritesmith
社区 Vite 适配版本，API 与 webpack-spritesmith 类似。
Vite 社区维护
svg-sprite-loader
将多个 SVG 合并为 sprite（symbol/use 模式），现代替代方案首选。
Webpack 推荐
趋势提醒： 大多数传统雪碧图工具（sprity、Glue）已停止维护。在 HTTP/2 普及的今天， 新项目建议直接使用 SVG sprite（svg-sprite-loader / vite-plugin-svg-icons） 或 Icon Font，仅在兼容老项目或位图图标场景下才使用 CSS Sprites。

2 构建工具配置示例

以下是真实项目中使用 webpack-spritesmith 和 Vite 自动生成雪碧图的配置示例（仅作参考，本演示不实际运行）。

webpack.config.js 
const path = require("path");
const SpritesmithPlugin = require("webpack-spritesmith");

module.exports = {
  // ...其他配置
  plugins: [
    new SpritesmithPlugin({
      // 源图目录：每个 png 是一个独立图标
      src: {
        cwd: path.resolve(__dirname, "src/icons"),
        glob: "*.png",
      },
      // 输出：合并后的雪碧图
      target: {
        image: path.resolve(__dirname, "src/assets/sprite.png"),
        css: path.resolve(__dirname, "src/assets/sprite.scss"),
      },
      // 样式模板中引用图片的路径
      apiOptions: {
        cssImageRef: "./sprite.png",
      },
      // 雪碧图布局：从上到下排列
      spritesmithOptions: {
        algorithm: "top-down",
        padding: 2,
      },
    }),
  ],
  // 让 webpack 识别 .png
  module: {
    rules: [
      { test: /\.png$/, type: "asset/resource" },
    ],
  },
};
生成的 sprite.scss（自动产出） 
/* 由 webpack-spritesmith 自动生成，请勿手动编辑 */
.icon-home {
  background-image: url('./sprite.png');
  background-position: 0px 0px;
  width: 64px;
  height: 64px;
}
.icon-user {
  background-image: url('./sprite.png');
  background-position: -64px 0px;
  width: 64px;
  height: 64px;
}
/* ... 更多图标 */
vite.config.ts（Vite + svg-icons，现代推荐） 
import { defineConfig } from "vite";
import { createSvgIconsPlugin } from "vite-plugin-svg-icons";
import path from "path";

export default defineConfig({
  plugins: [
    createSvgIconsPlugin({
      // 扫描 src/icons/svg 下所有 .svg
      iconDirs: [path.resolve(process.cwd(), "src/icons/svg")],
      // symbol id 格式：icon-文件名
      symbolId: "icon-[name]",
    }),
  ],
});

// 使用：自动生成 SVG sprite，配合 <use> 引用
// <svg><use href="#icon-home"/></svg>

3 四种图标方案横向对比

CSS Sprites / Icon Fonts / SVG / Base64 各有适用场景，下表从 8 个维度对比。

对比维度 CSS Sprites Icon Fonts SVG <symbol> Base64 内联
示例 
HTTP 请求数 1 次 1 次（字体文件） 1 次（可内联为 0） 0 次（嵌入 CSS）
矢量缩放 否（位图） 是 是 否（位图）
CSS 改色 否 单色（color） 是（fill/currentColor） 否
多色支持 是 否 是 是
Retina 适配 需 @2x 图 天然支持 天然支持 需 @2x 图
可访问性 需 aria-label 差（屏幕阅读器） 好（<title>） 需 aria-label
缓存复用 整图缓存 字体缓存 内联则不缓存 随 CSS 缓存
HTTP/2 收益 显著降低 不受影响 不受影响 不受影响
维护成本 高（需工具） 中 低 低
推荐场景 老项目、位图、游戏图集 单色图标库 现代项目首选 极小图（<4KB）
结论： 现代项目（HTTP/2 + Retina）首选 SVG <symbol> + <use>， 它兼具矢量缩放、CSS 改色、多色、可访问性、低维护成本等优势。CSS Sprites 在新项目中已不推荐， 但在维护老项目或处理位图图标时仍有价值。

解读：
  • CSS Sprites 把 40KB 压到 28KB（共享调色板），请求从 20 降到 1。

  • Base64 虽然请求数为 0，但体积膨胀 33%，且无法独立缓存，仅适合极小图。
  • HTTP/2 下"20 次请求"共用连接，握手开销可忽略，雪碧图的请求优势被削弱。

6 现代替代方案演示

在 HTTP/2 时代，以下三种方案已逐步取代 CSS Sprites。鼠标移到图标上查看交互效果。

1. SVG <symbol> + <use> 推荐

所有 SVG 合并为一个隐藏的 sprite，通过 <use href="#id"> 引用。 矢量无损、可用 CSS 改色（fill）、支持多色、可加 <title> 提升无障碍。

HTML 
<!-- 定义（页面只需一次） -->
<svg style="display:none">
  <symbol id="icon-home" viewBox="0 0 24 24">
    <path d="..." />
  </symbol>
</svg>

<!-- 使用 -->
<svg class="icon"><use href="#icon-home"/></svg>

2. Icon Font（图标字体）

   

把图标做成字体文件（woff2），用 <i class="fa fa-home"> 引用。 矢量缩放、单色可用 color 改色、体积小。但无法多色、可访问性差、字体加载有闪烁。

HTML 
<link rel="stylesheet"
      href="font-awesome.min.css">

<i class="fa fa-home"></i>
<i class="fa fa-heart"></i>

/* CSS 改色 */
.fa-home { color: #38bdf8; font-size: 24px; }

3. CSS 渐变背景（纯 CSS 图形）

用 linear-gradient / radial-gradient / conic-gradient 直接画出图形，零图片请求、矢量缩放、可用 CSS 动画。适合简单几何图形与占位符。

CSS 
.icon-gradient {
  width: 40px;
  height: 40px;
  border-radius: 8px;
  background: linear-gradient(
    135deg, #667eea 0%, #764ba2 100%
  );
}
迁移建议： 从 CSS Sprites 迁移到 SVG sprite 的工作量很小：
  1. 把 PNG 图标用工具（如 inkscape / figma）导出为 SVG；
  2. 用 vite-plugin-svg-icons 或 svg-sprite-loader 自动生成 sprite；
  3. 把 <span class="icon-xxx"> 替换为 <svg><use href="#icon-xxx"/></svg>；
  4. 改色用 fill: currentColor，跟随文字颜色。

7 选型决策流程

面对图标需求时，按以下问题逐步决策：

1
图标是否需要多色 / 渐变 / 复杂路径？
是 → SVG <symbol>　|　否 → 继续第 2 步
2
图标是否为照片级位图（如缩略图、游戏素材）？
是 → CSS Sprites 或独立图片 + HTTP/2　|　否 → 继续第 3 步
3
是否单色且数量较多（>20 个）？
是 → Icon Font　|　否 → 继续第 4 步
4
图标数量很少且体积极小（<4KB）？
是 → Base64 内联　|　否 → SVG <symbol>（默认首选）
5
是否在维护老项目（已有 PNG + HTTP/1.1）？
是 → 保留 CSS Sprites（用 webpack-spritesmith 自动化），逐步迁移 SVG
一句话总结： 新项目用 SVG，老项目用雪碧图过渡，单色图标库用 Icon Font，极小图用 Base64。 HTTP/2 环境下，"减少请求数"不再是首要目标，可维护性与可访问性更重要。
